Let's first see how maps can be used in Boogie.

\begin{verbatim}
procedure simple ()
{
    var y: int;
    var a: [int] int;
    var b: [int] int;
    
    y := a[0];       // select
    a := a[2 := 1];  // store and return a new map
    a[2] := 1;       // convinient syntax for the above

    // lambda expression
    a := (lambda x: int :: if x > 0 then y + x + x else y + x);
    b := (lambda x: int :: if x < 0 then y + x     else y + x + x);
  
    assert a == b;
}

\end{verbatim}

With the syntax \texttt{var m:[K] V}, we can declare a map
with key type \texttt{K} and value type \texttt{V}.
Using a map, we can select a value at a given key
and obtain an updated map by storing a new value at a corresponding key.

Interestingly, lambda expressions can be treated as maps.
Thus, a map behaves like a function that can be treated as a value and updated.
Importantly, it is \textbf{not} like a dictionary or an association list,
as those are usually finite mappings.

There are more complex map types.
They can have multiple keys, which we then call multi-arity maps.
This source [TODO ref] claims that they are discouraged though.
The key and value types can themselves be map types,
thus enabling nested map types.

\begin{verbatim}
procedure complex ()
{
    var y: int;
    var a: [int] int;
    var d: [ [int]int, int ] int;   // multi-arity
    var c: [ [int]int ] [int] int;  // curried
    
    c[a][y] := y;
    d[a, y] := y;
}
\end{verbatim}


Maps can also be polymorphic, i.e. have a type parameter:

\subsection{VC axioms changes}
\label{sec:vc-axioms-changes}

Supporting maps, changes the VC as discussed in \ref{sec:vc-structure}.
It introduces more function and axioms regarding map types and map behaviours.
Remember, to be able to use the conclusion of the VC, the weakest precondition,
we need to fulfill the antecedant first.
These gives us some formal goals, our map semantics need to fulfill.

\[
\forall
\underbrace{\textit{VC quantifiers}}_{\substack{
  \text{type encoding parameters,}\\
  \text{functions, variable values}\\
  \textbf{map types \& functions}}}
\mathpunct{.}\;
\Bigl(
\underbrace{\textit{VC assumptions}}_{\substack{
  \text{type encoding,}\\
  \text{fun./var./prog. axioms}\\
  \textbf{map axioms}}}
\Longrightarrow
\textit{WP}
\Bigr)
\]

These differ greatly with different typeencodings.
Since the preceeding work was built on polymorphic typeencoding,
we will be focusing on its encoding of the maps.


\subsubsection{VC Map Axioms}
\label{sec:vc-map-axioms}

\newcommand{\opApp}[1]{\operatorname{#1}}
\newcommand{\opAppI}[2]{\operatorname{#1} \ {#2}}
\newcommand{\opAppII}[3]{\operatorname{#1} \ {#2} \ {#3}}
\newcommand{\opAppIII}[4]{\operatorname{#1} \ {#2} \ {#3} \ {#4}}

Here are the map relevant parts of the VC:

We have a sorts $U, T$, where $U$ represents values, and $T$ represents types.
We also have a function $type : U \rightarrow T$ that gives the type of a value.
Then are also the functions:
\footnote{Boogie generates even more function, for each arity type of map,
$MapType0$ just refers to the single-arity maps.}
\begin{itemize} 
  \item $MapType0Select : (U, U) \rightarrow U$, selecting from a map
  \item $MapType0Store : (U, U, U) \rightarrow U$, updating a map
  \item $MapType0Type : (T, T) \rightarrow T$, constructing a map type from two types
  \item $MapType0TypeInv0 : T \rightarrow T$, extracting the key type
  \item $MapType0TypeInv1 : T \rightarrow T$, extracting the value type
\end{itemize} 

Notably, these function apply to \emph{all} values, not just map values.

These functions need to fulfill the following axioms: \footnote{simplified for clarity}

\begin{align*}
  & \forall\ k\ v.\ MapType0TypeInv0\ (MapType0Type\ k\ v) = k\\
  & \forall\ k\ v.\ MapType0TypeInv1\ (MapType0Type\ k\ v) = v
\end{align*}

These guarantee that $MapType0TypeInv0$ and $MapType0TypeInv1$ are actually
inverse functions of the map type constructor $MapType0Type$.

\begin{equation}
  \label{ax:map-select-typesafe}
  \begin{aligned}
    \forall \ m \ k. \ 
    type \ m = MapType0Type \ (type \ k) \ (MapType0TypeInv1 \ (type \ m)) \\
    \implies  type \ (MapType0Select \ m \ k) = MapType0TypeInv1 \ (type \ m)
  \end{aligned}
\end{equation}

% TODO shorter explanantion?
This says, that if we have a value $k$ of type $K = type \ k$
and a map $m$ of type $MapType0Type \ K \ V$, where $V = MapType0TypeInv1 \ m$
then the result of the select $MapType0Select \ m \ k$ has the type at the map claims,
$type \ (MapType0Select \ m \ k) = V$.
In short, if we select from a map with the correct key type,
then the result is of the corresponding value type.

\newcommand{\MapType}[2]{\operatorname{MapType0} \ {#1} \ {#2}}
\newcommand{\MapTypeKey}[1]{\operatorname{MapType0TypeInv0} \ {#1}}
\newcommand{\MapTypeVal}[1]{\operatorname{MapType0TypeInv1} \ {#1}}

\begin{align*}
  &\forall \ m \ k \ v. \\
  &\type m = \MapType {(\type k)} {(\type v)} \\
  &\implies  \type(\MapTypeStore m k v) = \MapType {(\type k)} {(\type v)}
\end{align*}

The store operation returns a map of the same type.

\begin{equation}
  \label{ax:map-update-strong}
  \begin{aligned}
    &\forall \ m \ k \ v.\\
    &\type v = \MapTypeVal {(\type m)} \\
    &\implies  \MapTypeSelect {(\MapTypeStore m k v)} k = v
  \end{aligned}
\end{equation}

This shows, that updating at a position works.

\begin{align*}

\end{align*}

This shows, that the store at a position, does not affect the other positions.


\subsubsection{Our VC antecedant change}
\label{sec:our-vc-change}

The current axiom, talking about the map update \ref{ax:map-update-strong},
is too strong for our semantics,
that relies on our stratification solution, explained leater in section \ref{sec:stratification-solution}.
The main problem is, that this axiom has to hold for every value $m$,
even is it is not a map, and every value $k$,
even if its type does not match the key type of $m$.
That is why we decided to weaken the axiom:

\begin{equation}
  \label{ax:map-update-weak}
  \begin{aligned}
    &\forall \ m \ k \ v.\\
    &\type v = \MapTypeVal {(\type m)} \\
    &\type m = \MapType {(\type k)} {(\type v)}\\
    &\implies  \MapTypeSelect {(\MapTypeStore m k v)} k = v
  \end{aligned}
\end{equation}

With $\type m = \MapType {(\type k)} {(\type v)}$ we guarantee,
that $m$ is a map, and that the store is applied with the corresponding types.


\subsection{Where to start}
\label{sec:where-to-start}
In the Boogie semantics, maps are treated as first-class values.
Consequently, the initial step in supporting them involves extending the existing value datatype.
The current definition for literals and values is as follows:

\begin{verbatim}
datatype lit =  LBool bool  | LInt int | LReal real

(* The values (and as a result the semantics) are parametrized by the carrier type 'a
for the abstract values (values that have a type constructed via type constructors) *)

datatype 'a val = LitV lit | AbsV (the_absv: 'a)
\end{verbatim}

To incorporate Boogie Maps, we must introduce a new constructor to this datatype:

\begin{verbatim}
datatype 'a val = LitV lit | AbsV (the_absv: 'a) | MapV ?
\end{verbatim}

The question now is, what should we put in place of \texttt{?}?

\subsection{Design Challenges: Cantor's Theorem}
\label{sec:cantor}

A naive approach to representing map values would involve a direct recursive definition:

\begin{verbatim}
datatype 'a val = LitV lit | AbsV (the_absv: 'a) | MapV "'a val => 'a val"
\end{verbatim}

However, Isabelle/HOL rejects this definition.
This is not a technical limitation of the tool, but a fundamental logical constraint.
Such a definition contradicts Cantor's Theorem,
which states that the cardinality of a function space $A \implies A$ is strictly greater than the cardinality of $A$.
Since the constructor \texttt{MapV} would imply an injective function from the function space back into the value type,
it would lead to a logical inconsistency.

Notably, the issue arises specifically from the recursive occurrence of \texttt{'a val} in the domain (the left-hand side) of the function.
While a constructor like \texttt{bool => 'a val} is permissible, \texttt{'a val => bool} is not.

\subsection{The Stratification Solution}
\label{sec:stratification-solution}
To resolve the conflict between recursion and cardinality, we propose an ``explicit unrolling'' of the recursion on the domain.
This creates a hierarchy of values:

\begin{itemize}[leftmargin=1.5em]
  \item Level 0: Contains only primitive values: literal and abstract values.
  \item Level 1: Maps that accept Level 0 keys and return Level 0 or Level 1 values.
  \item Level 2: Maps that accept Level 0 or Level 1 keys and return Level 0, Level 1 or Level 2 values.
  \item Level $n$: Maps where keys are of Level $\le n-1$ and return values are of Level $\le n$.
\end{itemize}

This approach requires the user to specify the required nesting depth when defining the semantics.

To ensure the semantics can represent complex, nested Boogie map types,
it is essential that each level $n$ is cumulative, incorporating all preceding levels.
This allows the model to support maps where the keys or values are themselves maps of a lower level.
For instance, representing types such as
\texttt{[int][[int]int]int} (a map of level 2, with keys of level 0, returning values of level 2) or \texttt{[[[int]int]int]int} (a map of level 3, with keys of level 2, returning values of level 0).

\subsubsection{Program dependent semantics}

A notable characteristic of this approach is that the value semantics are program-dependent.
Currently, the implementation is fixed at three levels of nesting.
While this serves as a practical limitation for the current prototype,
it is theoretically extensible to any finite depth required by a target Boogie program.
This depth has to be determined, such that we are able to define the types and functions of the map semantics properly.
That means depending on the input program, we get a different semantics.

\subsubsection{Datatype Construction}
\label{sec:datatype-construction}
The stratified hierarchy could be implemented by defining discrete datatypes for each level:

\begin{verbatim}
datatype 'a val0 = LitV0 lit | AbsV0 (the_absv: 'a)
datatype 'a val1 = MapV1 "'a val0 => ('a val1 + 'a val0)"
datatype 'a val2 = MapV2 "('a val1 + 'a val0) => ('a val2 + 'a val1 + 'a val0)"
datatype 'a val3 = MapV3 "('a val2 + 'a val1 + 'a val0)
  => ('a val3 + 'a val2 + 'a val1 + 'a val0)"
\end{verbatim}

To minimize redundancy and improve maintainability, we introduce a helper datatype, \texttt{'k L},
which represents a functional map layer.
We then construct the hierarchy using type synonyms:

\begin{verbatim}
(* A helper datatype, that takes a domain as a type parameter
   and gives a function from the domain to the domain and itself *)
datatype 'k L = FunL "'k => 'k L + 'k"

(* A datatype representing just primitive values *)
datatype 'a val0 = LitV0 lit | AbsV0 (the_absv: 'a)

(* Instantiate different map levels *)
type synonym 'a val1 = "'a val0 L"
type_synonym 'a val2 = "('a val1 + 'a val0) L"
type_synonym 'a val3 = "('a val2 + 'a val1 + 'a val0) L"
type_synonym 'a val3210 = "'a val3 + 'a val2 + 'a val1 + 'a val0"
\end{verbatim}

To integrate these stratified levels into the core semantics,
we extend the primary \texttt{val} datatype with a generic \texttt{MapV} constructor.
By utilizing a second type parameter, \texttt{'m}, we decouple the value definition from the specific map implementation:

\begin{verbatim}
datatype ('a, 'm) val = LitV lit | AbsV (the_absv: 'a) | MapV 'm
\end{verbatim}

This allows us to instantiate the semantics with a specific number of levels:

\begin{verbatim}
(* Instantiate values with three map levels *)
type synonym 'a val321  = "'a val3 + 'a val2 + 'a val1"
type synonym 'a valn = "('a, 'a val321) val"
\end{verbatim}

Notably, \texttt{'a val0} is excluded from the \texttt{MapV} constructor,
as these represent primitive literals and abstract values which are semantically distinct from map-based values.

However, establishing the datatype is only the foundational step;
we must subsequently define the functional operations and verify that they satisfy the required Boogie array axioms.
See Section~\ref{sec:vc-map-axioms}.

\subsubsection{Type of Values}
\label{sec:type-of-values}
A critical component of the value semantics, particularly for map operations such as \texttt{store} (Section~\ref{sec:store}) and well-formedness (Section~\ref{sec:wf-definition}),
is the \texttt{type\_of\_val} function.
This function determines the corresponding Boogie type for any given value within the semantics.
To support Boogie maps, the existing type definition must be extended to include a map type constructor:

\begin{verbatim}
datatype prim_ty
 = TBool | TInt | TReal

type_synonym tcon_id = string (* type constructor id *)

datatype ty
  = TVar nat | (* type variables as de-bruijn indices *)
    TPrim prim_ty | (* primitive types *)
    TCon tcon_id "ty list" (* type constructor *) |
    TMap ty ty (* maps *)
\end{verbatim}

As the current implementation is limited to single-arity maps
\ref{sec:goals-to-support}, the \texttt{TMap} constructor requires only two arguments:
the key type and the value type.
Supporting multi-arity maps would necessitate representing the keys as a list of types,
but the current dual-type representation is sufficient for our defined goals (Section~\ref{sec:goals-to-support}).

\paragraph{Type Ambiguity}
A challenge arises when implementing type extraction for map values.
Because our functional modeling of maps is so general,
a single internal Isabelle function could represent multiple distinct Boogie map types.
For instance, a map of type \texttt{val1} might contain a function mapping integers and booleans to integers and booleans,
making it impossible to distinguish from the outside whether it represents
a \texttt{[int]int}, \texttt{[bool]bool}, or \texttt{[int]bool} map.

To resolve this ambiguity, we explicitly store the intended key and value types within the map datatype:

\begin{isabellebody}
\isakeywordONE{datatype}\isamarkupfalse%
\ {\isacharprime}{\kern0pt}p\ L\ {\isacharequal}{\kern0pt}\ FunL\ {\isachardoublequoteopen}{\isacharprime}{\kern0pt}p\ {\isasymRightarrow}\ {\isacharprime}{\kern0pt}p\ L\ {\isacharplus}{\kern0pt}\ {\isacharprime}{\kern0pt}p{\isachardoublequoteclose}\ ty\ ty\isanewline
\isanewline
\end{isabellebody}


While this ensures that every map value carries explicit type information,
it introduces a potential discrepancy:
there is no inherent semantic connection between the stored types and the actual behavior of the Isabelle function.
For example, a value could be tagged as \texttt{[bool]bool} while the underlying function operates on integers.
This consistency requirement is addressed later through the definition of well-formedness invariants (Section~\ref{sec:wf-definition}).

\paragraph{Accessor Functions for Map Types}
With the inclusion of explicit type tags,
we can define helper functions to extract the constituent types from a map type.
These functions are essential for the subsequent implementation of map operations:

\begin{verbatim}
fun key_ty where "key_ty (TMap tk _) = tk" | "key_ty _ = undefined"
fun val_ty where "val_ty (TMap _ tv) = tv" | "val_ty _ = undefined"
\end{verbatim}

Detailed definitions for the full \texttt{type\_of\_val} logic are provided in Appendix~\ref{sec:appendix}.

These function will also play a crucial part in fulfilling the VC map axioms \ref{sec:vc-map-axioms},
especially regarding $\opApp{MapType}, \opApp{MapType0TypeInv0}, \opApp{MapType0TypeInv1}$.


\subsubsection{Constrained Types}
\label{sec:constrained-types}

The current datatype definition is too permissive,
and does not fulfill our required map axioms from \ref{sec:vc-map-axioms}.
We will need to restrict them, with some additional well-formedness properties, $\opApp{wf}$,
that the datatype needs to fulfill.
Using these properties, we are able to create a new 'constrained' type,
using using \texttt{typedef} in Isabelle. % TODO ref

\begin{verbatim}
definition wf_map_set :: "'a val321 set" where
"wf_map_set = {m. wf (MapV m)}"

typedef (overloaded) 'a wf_maps = "wf_map_set :: 'a val321 set"
proof
  (* showing some witness, such that wf_maps is not empty *)
qed

type_synonym 'a wf_val = "('a, 'a wf_maps) val"
\end{verbatim}

The question remains, how to define \texttt{wf}.

\subsubsection{Closed}
\label{sec:closed}

If we have a constrained type, and use operations like select/store,
the result should be constrained again, i.e. fulfill our desired properties.
Thus, one desired property is \texttt{(ALL k. wf k --> wf (selectImpl m k))}.
\footnote{Assuming we have a suitable definition of \texttt{selectImpl}, which will be introduced in Section~\ref{sec:select}}

We can see that that the definition of \texttt{wf} is dependent on \texttt{wf} of other values.
For a recursive definition, we would need to guarantee termination,
e.g. applying \texttt{wf} only on 'smaller' cases.
This is not that easy in this case, as \texttt{selectImpl m k} might be any other value.
Fortunately there are \texttt{inductive} definitions in Isabelle/HOL %TODO ref,
allowing us to define \texttt{wf} values based on previously determined \texttt{wf} values.

The recursive \texttt{wf} call is not allowed in negative position, which is a problem, since:
\texttt{(ALL k. wf k --> wf (selectImpl m k))} is \texttt{(ALL k. ¬ wf k OR wf (selectImpl m k))}.
This also makes intuitively sense, since we are only able to show that values are \texttt{wf},
but not that there are \texttt{¬wf}.
So our solution is to make the property stronger, by dropping the antecedant:
\texttt{(ALL k. wf (selectImpl m k))},
enforcing that the result is always \texttt{wf}, no matter if the key is or not.

For store we do not need an additional property, as we can derive the property:
\texttt{(ALL k v. wf m AND wf k AND wf v --> wf (storeImpl m k v))}.

This leads to our work-in-progress definition of:

\begin{isabellebody}
\isakeywordONE{inductive}\isamarkupfalse%
\ wf\ \isakeywordTWO{where}\isanewline
\ \ \ \ wfLitV{\isacharcolon}{\kern0pt}\ {\isachardoublequoteopen}wf\ {\isacharparenleft}{\kern0pt}LitV\ v{\isacharparenright}{\kern0pt}{\isachardoublequoteclose}\ {\isacharbar}{\kern0pt}\ wfAbsV{\isacharcolon}{\kern0pt}\ {\isachardoublequoteopen}wf\ {\isacharparenleft}{\kern0pt}AbsV\ v{\isacharparenright}{\kern0pt}{\isachardoublequoteclose}\ {\isacharbar}{\kern0pt}\isanewline
\ \ \ \ wfMapV{\isacharcolon}{\kern0pt}\ {\isachardoublequoteopen}{\isasymlbrakk}\ {\isacharparenleft}{\kern0pt}{\isasymforall}k{\isachardot}{\kern0pt}\ wf\ {\isacharparenleft}{\kern0pt}selectImpl\ m\ k{\isacharparenright}{\kern0pt}{\isacharparenright}{\kern0pt}

\end{isabellebody}

Where we have a base-case, saying that the primitive values are well-formed,
and a map case, where we can have our additional properties as the antecedants for implying \texttt{wf}.

\subsubsection{Type Safety}
\label{sec:type-safety}
From the VC map axiom~\ref{ax:map-select-typesafe},
we need the property that if we select from a map using a key of the key type of the map,
then the result is of the value type of the map.
So in Isabelle we want something like this:

\begin{verbatim}
(ALL k. wf k AND type_of_val k = key_ty (type_of_val m)
--> type_of_val (selectImpl m k) = val_ty (type_of_val m))
\end{verbatim}

%[TODO mention why it is hard to do that structurally using datatypes only?]

We will actually enforce a stronger condition:

\begin{verbatim}
(ALL k. type_of_val (selectImpl m k) = val_ty (type_of_val m))
\end{verbatim}

One reason is, again, that \texttt{wf} may not appear in a negative position
(in this case, on the left hand side of an implication).
Then it will be a derived property, after removing \texttt{wf}
and in combination with the other properties.
We think, it is more convenient to enforce this explicitly,
such that we can have a stronger guarantee,
for cases where we need to build nested map and prove its well-formedness.

We extend our definition of \texttt{wf} accordingly:

\subsubsection{No Duplicate representation}
\label{sec:no-duplicate}

As discussed in Section~\ref{sec:goals-to-support},
we want our maps to be extensional.
More concretely, this means, that we want there to be a unique Isabelle representation for each Boogie map.
This means, that if two maps values agree for each key,
then they should be represented by the same Isabelle representation.
The current definition has two problems fulfilling this property.

\paragraph{Problem 1: Low-level map as high level value}
Let's say we have a \texttt{[int]int} map in Boogie.
We want to cononically represent this in Isabelle as a value of level 1, i.e. \texttt{'a val1}.
Unfortunately, there is nothing stopping us from representing it as a map on level 3, i.e. \texttt{'a val3},
as it has \texttt{'a val0} in its domain and codomain.
Therefore, we have different ways to represent the same thing.

\paragraph{Solution: Type Hierarchies}
To avoid this we make sure that the map types on level $n$,
need to either have their domain on level $n-1$ or their codomain on level $n$.

\paragraph{Problem 2: Function behaviour outside the domain}
Let's say, we have a map in Boogie \texttt{(lambda x: int :: x+1)}.
In its embedding in Isabelle the domain is not constrained to integers only
So we might have two maps, that both capture the same behaviour on integers,
but might differ on boolean for example.
Therefore, we also want to make sure that the all the maps, that capture the same behaviour in Boogie,
also behave the same outside their intended domain.
The easiest thing to do here is to enforce a sentinel value,
for the keys outside the intended domain.
The values outside the intended domain, are not only the ones with the wrong type,
but also all the not well formed values.
Therefore, we want to enforce something like this:

\begin{isabellebody}
{\isasymforall}k{\isachardot}{\kern0pt}\ {\isacharparenleft}{\kern0pt}{\isasymnot}wf\ k\ {\isasymor}\ type{\isacharunderscore}{\kern0pt}of{\isacharunderscore}{\kern0pt}val\ k\ {\isasymnoteq}\ key{\isacharunderscore}{\kern0pt}ty\ {\isacharparenleft}{\kern0pt}type{\isacharunderscore}{\kern0pt}of{\isacharunderscore}{\kern0pt}val\ m{\isacharparenright}{\kern0pt}{\isacharparenright}{\kern0pt}\ {\isasymlongrightarrow}\ {\isacharparenleft}{\kern0pt}selectImpl\ m\ k{\isacharparenright}{\kern0pt}\ {\isacharequal}{\kern0pt}\ sentinel?{\kern0pt}
\end{isabellebody}

The question is, what kind of \texttt{sentinel?} value we need?
As we have seen in Section~\ref{sec:type-safety}, we are not able to cleanly case distinct on \texttt{wf k},
as \texttt{wf} may not appear in a negative position.
Therefore, we might have a \texttt{k}, where we are not sure if it is \texttt{wf k} or \texttt{¬wf k},
and therefore not sure if \texttt{type\_of\_val (selectImpl m k) = val\_ty (type\_of\_val m)} needs to hold or \texttt{type\_of\_val (selectImpl m k) = sentinel?}.
If the \texttt{sentinel?} is of type \texttt{val\_ty (type\_of\_val m)}, then both conditions hold anyway, and we have no problem.
Fortunately, we have a function \texttt{val\_of\_type}, that gives us a value of the given type,
which we can use for our sentinel value:\\
\texttt{sentinel? = val\_of\_type (val\_ty (type\_of\_val m))}

\label{sec:select}
